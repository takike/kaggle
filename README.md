# ROGII – Wellbore Geology Prediction (Geosteering / TVT)

A complete, runnable solution for the Kaggle competition
[**ROGII – Wellbore Geology Prediction**](https://www.kaggle.com/competitions/rogii-wellbore-geology-prediction)
($50K, Featured Code Competition).

> **Task.** For each ~1‑ft section along a *horizontal (lateral)* well, predict
> the **True Vertical Thickness (TVT)** — the bit's stratigraphic position relative
> to a marker. **Metric: RMSE.** Inputs are the lateral well trajectory + Gamma‑Ray
> (GR) log, plus a vertical **typewell** reference log GR(TVD). Public‑LB leaders sit
> around RMSE ≈ 9–12.

This repo treats the problem as what it physically is — a **geosteering log‑correlation**
problem — rather than as plain tabular regression, and that framing is where almost
all of the accuracy comes from.

---

## Why this is *not* a plain tabular problem

A single GR value is ambiguous: many depths share the same GR. The signal lives in
the **shape** of the GR curve as the lateral undulates through the layers, matched
against the typewell. The core inference is therefore an **alignment**:

> find the slowly‑varying *structural shift* `delta(MD)` so that a short window of the
> lateral GR best matches the typewell GR at depth `tvd − delta`; then
> **`TVT = (tvd − delta) − datum`**.

`delta` (the structure) is smooth along the hole; TVT changes quickly as the well
goes up/down through beds. We solve the alignment with a **dynamic program (Viterbi)**
whose transition cost is L1 in `delta` (computed in O(states) per step via an L1
distance transform), so it is exact and fast.

### Pipeline

```
data ─▶ geosteering alignment (ensemble) ─▶ features ─▶ LightGBM residual model ─▶ blend ─▶ along-hole smoothing
```

1. **Aligner** (`src/align.py`) – windowed GR matching + Viterbi smoothing of `delta`.
2. **Zone prior** – a soft pull of TVT toward the target‑zone centre breaks the
   quasi‑periodic *marker ambiguity* that otherwise locks a well one GR cycle off.
3. **Aligner ensemble** (`src/pipeline.py`) – 5 diverse `(lam, window, prior)` configs
   averaged. Different configs lock the cycle slightly differently, so the mean is
   robust to the occasional bad well; their **disagreement (std)** becomes a
   confidence feature.
4. **LightGBM residual model** (`src/features.py`, `src/pipeline.py`) – learns
   `TVT − tvt_align` from trajectory geometry, multi‑scale GR texture and alignment
   confidence. **GroupKFold by well** (a well is never split) mirrors the held‑out‑wells
   leaderboard and prevents leakage.
5. **Along‑hole median smoothing** – TVT is spatially coherent.

## Results (held‑out‑wells, GroupKFold OOF)

Generated by `experiments/run_ablation.py` on physics‑faithful synthetic data,
averaged over 5 seeds (see [`experiments/ablation_results.md`](experiments/ablation_results.md)):

| Stage | mean RMSE |
|---|---|
| naive: global mean | 16.59 |
| align: single, no prior | 4.03 |
| align: **ensemble** | 3.36 |
| ML: LightGBM direct (no alignment) | 4.34 |
| ML: ensemble + LightGBM residual | 3.25 |
| **FINAL: + along‑hole smoothing** | **3.25** |

**Read‑outs of the experimentation:**
- The **alignment is the whole game**: it takes RMSE from 16.6 → ~3.4. A LightGBM
  trained on raw features *without* alignment only reaches 4.34 — i.e. the geosteering
  framing beats brute‑force tabular ML.
- A naïve global zone prior *hurts* (4.03 → 4.79): it rescues wells that lock a cycle
  off but biases wells whose correct lock is genuinely offset. **Ensemble diversity**
  (mixing no‑prior and graded‑prior aligners) is what actually delivers — it lowered
  both the mean (4.03 → 3.36) and the worst‑seed case (6.39 → 4.40).
- ML refinement + smoothing add a final, modest gain (3.36 → 3.25). On the *real* data
  (noisier logs, partial typewell coverage, faults) the residual model and the
  confidence features are expected to matter more than they do on clean synthetic data.

> Absolute numbers are on synthetic data and are **not** directly comparable to the
> real leaderboard — the synthetic GR template is cleaner than real logs. They
> measure the *relative* value of each component, which is the point.

## Real competition data (validated)

The real Kaggle data was later obtained and the solution adapted to its actual
schema (`src/real_io.py`, `src/predict_real.py`, `src/submit_real.py`). The real
task is a **predict‑ahead geosteering** problem:

- Each case = `<id>__horizontal_well.csv` (MD, X, Y, Z, GR, `TVT_input`, +train‑only
  `TVT` & formation tops) and `<id>__typewell.csv` (TVT, GR, Geology).
- `TVT` is known only up to a **Prediction Start (PS)** point (`TVT_input`, a
  contiguous "heel"); you predict TVT for the rest of the lateral (the "toe").
  Metric: pooled RMSE of `true_TVT − pred_TVT` over toe rows.
- The structural offset **`C = TVT + Z`** varies smoothly along MD (≈ the geologic
  dip), drifting ~165 ft on average across the toe — so you must *follow* the
  layers ahead by correlating toe GR to the typewell.

**Algorithm** (`predict_case`): build a reference `GR(TVT)` from the typewell
(affinely **calibrated to the horizontal GR scale on the heel overlap**) plus the
heel's own higher‑resolution GR; then a **Viterbi DP whose states are the deviation
of `C` from the heel's trend line** (so the search is bounded/centred and never
clips, no matter how far `C` drifts), with windowed absolute‑GR matching as
emission, `|Δdeviation|` (curvature) as transition cost, and the known heel pinned
as a hard anchor.

**Held‑out validation** (250 train cases, true TVT known — `experiments/real_results.md`):

| Method | pooled toe‑RMSE |
|---|---|
| trend‑only (extrapolate heel dip) | 43.48 |
| **geosteering aligner** | **19.81** (per‑case median **7.59**) |

Key lessons from the real‑data iteration:
- **States as deviation‑from‑trend** was the decisive fix: absolute‑`C` states
  clipped when structure drifted past the grid (one well: RMSE 152 → 62).
- **Absolute GR matching beats shape‑normalised** here — the heel calibration makes
  GR levels reliable and informative (shape‑norm pooled 42 vs 19).
- The pooled score is **tail‑dominated**; a uniform blend toward the trend doesn't
  help (the trend is worse on the hard cases), so the gap to LB leaders (~9) needs a
  *confidence‑aware* / learned correction.

```bash
# Run on the real data (after `kaggle competitions download -c rogii-wellbore-geology-prediction`)
python experiments/validate_real.py 250          # held-out toe-RMSE
python src/submit_real.py --data data_real --split test --out submission.csv
```

> Submission to the live LB is not possible here: ROGII is a **Code Competition**
> (hidden test, notebook only); direct CSV upload returns 403. The held‑out numbers
> above are the honest proxy.

### Closing the gap to LB leaders (~9)
- **LightGBM residual + confidence correction — explored, not adopted.**
  Built in `src/real_ml.py` (per‑toe‑row residual on relative/confidence features,
  GroupKFold by case). Honest result: at best **~1% pooled gain while degrading the
  median case** — no blend weight improves both (see
  [`experiments/ml_residual_results.md`](experiments/ml_residual_results.md)). On the
  ~80% well‑aligned cases the residual is near‑zero noise, so a global model hurts
  good cases faster than it fixes the idiosyncratic tail. The aligner stays as the
  production prediction.
- **Offset wells (most promising next).** The task notes neighbouring wells share
  structural dip; use X/Y to borrow the *toe* dip from nearby wells as a better prior
  for the aligner (kriging of `C`), rather than extrapolating the heel trend alone —
  more likely to help the tail than a post‑hoc residual.

## Running on the **real** Kaggle data

The pipeline never hard‑codes column names — everything routes through
[`src/config.py`](src/config.py).

1. Download the competition data into `data/` (e.g. `kaggle competitions download -c
   rogii-wellbore-geology-prediction`).
2. Inspect the real CSV headers and edit the right‑hand strings in `config.COLS`
   (`well`, `md`, `tvd`, `x`, `gr`, `target`, typewell columns, `id`) and `config.FILES`
   to match. If `tvd`/`x` are not provided directly they can be derived from the
   trajectory (MD, inclination, azimuth) via the minimum‑curvature method.
3. Set the zone prior centre/scale from the *training* TVT distribution
   (`tvt_center ≈ train.tvt.mean()`, `tvt_scale ≈ train.tvt.std()`), passed through
   `align_kw`.
4. Generate a submission:
   ```python
   from pipeline import make_submission, load_real
   train, tw = load_real("data")
   import pandas as pd; test = pd.read_csv("data/test.csv")
   make_submission(train, test, tw, out_path="submission.csv")
   ```

### Where to find more accuracy on the real data
- **Typewell handling.** Real comps often ship multiple typewells / offset wells —
  align against the nearest, or build a field‑wide structural surface and **krige**
  `delta` across wells (a known strong approach on this competition).
- **More logs.** If resistivity/density are present, stack them into the matching cost
  (multi‑curve correlation) — far more discriminative than GR alone.
- **Confidence‑aware blending.** Use `tvt_align_std` / `align_cost` to switch between
  alignment and ML where the lock is weak.
- **Tune `lam`/window** to the real GR sampling rate and bed thicknesses.

### Note on development order
The synthetic harness was built first (the environment initially had no Kaggle
access) to make genuine end‑to‑end trial‑and‑error possible. The real data was then
obtained and the solution adapted to its actual schema in `src/real_io.py` /
`predict_real.py` / `submit_real.py`, and validated on held‑out train cases
(`experiments/real_results.md`). The synthetic generator reproduces the same
geosteering forward model and remains the fast iteration loop.
